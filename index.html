05 — Methodology
Agile vs. Waterfall

Waterfall — sequential, preferred for ERP with stable requirements and hard deadlines. Clear phases with defined sign-offs and deliverable gates.

Agile / Scrum — sprint-based iterations for module rollouts where requirements evolve. Faster feedback cycles, early delivery of working increments.

Odoo principle: "Standard First" — always configure before customizing.

3-Tier Architecture

Strict three-layer architecture separating data, logic, and presentation. PostgreSQL is the only supported database layer.

Layer 1: PostgreSQL
→ Relational data, ORM tables
→ Only supported RDBMS

Layer 2: Python (ORM + Logic)
→ Models, methods, wizards

Layer 3: OWL / XML / JSON-RPC
→ Forms, lists, kanban, QWeb
→ Browser via JSON-RPC

03 / 06
Delegation Inheritance (_inherits)

Creates a parent-child table relationship — the child model embeds the parent, reusing data synchronously without database redundancy. Used in Education CRM: Student → res.partner.

# _inherits = table delegation
education.student:
_inherits = {
'res.partner': 'partner_id'
}
# Inherits: name, phone, email
# FK only — no duplication
# Example: res.users → res.partner

Security — 4 Layers

Hierarchical security. A user must pass all four layers to access or modify a record. Critical for multi-department access control design.

Menu Visibility

View / Field Access
Field-level: invisible, readonly, required by group

01
Procurement & Dynamic Approval
Purchase · Inventory · QC
Purchase
Purchase Request→ Approval Engine→ Purchase Order→ QC Check→ Stock + Cost Update
Problem

A bookstore using centralized purchasing had rigid approval workflows with no quality inspection on goods receipt. POs were created without budget validation — causing unauthorized spending, duplicate orders, and inventory discrepancies with no audit trail.

Solution

Designed a dynamic multi-level approval matrix in Odoo Purchase. Approval routing adapts automatically based on total_amount threshold. approval.config table allows defining approval levels by value. QC step with security tag verification before stock confirmation.

Key Features
  • PR → PO with full state tracking
  • approval.config: amount-based level routing
  • L1: <10M VND → done; ≥10M → L2 director
  • QC: security tag + exterior inspection step
  • Auto inventory + unit cost update from PO
  • Rejection with mandatory comment trail
Database Design
purchase.request date_required, total_amount: Monetary ← triggers level state: draft|submitted|approve1|approve2|done|rejected approval_config_id → approval.config purchase.request.line request_id → purchase.request (One2many) product_id, quantity, estimated_cost qc_status: pending | pass | fail
Business Impact

Eliminated unauthorized procurement by enforcing approval gates before any budget commitment. Full audit trail from PR creation through QC and receipt. QC gate prevents defective goods from entering inventory. The entire PR-to-receipt flow is traceable in one Odoo chain with zero manual handoffs.

02
B2B Master Contract Management
Sales · Partial Delivery · Invoicing
B2B Sales
Master Contract→ Sub-Orders→ Partial Delivery→ Delivery Invoice→ Utilization Tracking
Problem

B2B clients commit to 1,000 units but require call-off deliveries in multiple batches. Invoicing the full amount on partial delivery caused payment disputes and debt reconciliation errors. No system tracked contract utilization in real time.

Solution

Master Order as immutable contract + sub-orders for each call-off. Computed remaining_qty prevents over-ordering. Backorder mechanism handles partial fulfillment. Invoice policy = 'delivery' ensures billing only matches actual goods delivered.

Key Features
  • sale.master.order → sale.master.order.line
  • remaining_qty = committed_qty − delivered_qty
  • Immutable master contract post-confirmation
  • Backorder auto-clones partial fulfillment
  • Constraint: block sub-order qty > remaining
  • Invoice = qty_delivered − qty_invoiced only
Algorithm — Computed Fields & Constraints
sale.master.order.line: committed_qty: Float delivered_qty = fields.Float( compute='_compute_delivered', store=True ) # sum of non-cancelled sub-order lines remaining_qty = committed_qty - delivered_qty # Constraint: block over-delivery @api.constrains('qty_ordered') def _check_remaining(self): if qty_ordered > master_line.remaining_qty: raise ValidationError("Exceeds contract volume") # Invoice policy invoice_policy = 'delivery' ← on sale.order
Business Impact

Sales and finance gained real-time contract utilization visibility. Eliminated over-invoicing risk — all invoices tied to confirmed deliveries. Account managers proactively alert clients near contract limits, reducing disputes and improving trust.

03
Education CRM — Admissions Pipeline
CRM · K-12 · Delegation Inheritance · Financials
CRM / Custom
Parent Lead→ Application→ Level Assessment→ Sibling Discount→ Enrollment
Problem

Manage parents (payers) separately from students (learners). Admissions pipelines differ completely between levels (Nursery vs. Secondary). One mother enrolling two children at different levels required duplicate contact records — causing data inconsistency.

Solution

Delegation Inheritance links education.application to res.partner for payer data reuse without duplication. Dynamic UI via attrs hides/shows fields by education_level. Sibling discount auto-computed. Payment + test score validation blocks premature enrollment.

Key Features
  • education.parent + education.application via phone
  • _inherits → res.partner: no contact duplication
  • Nursery: show vaccination book, hide test_score
  • Secondary: test_score required (attrs)
  • is_sibling computed → auto sibling discount
  • Block "Enrolled" if paid_amount = 0 or test_score = 0
  • Prorated fee on withdrawal (per month studied)
  • Auto Student_ID on sale confirmation
Data Structure & Dynamic UI
education.application: _inherits = {'res.partner': 'partner_id'} # Inherits: name, phone, email, address education_level: Selection[nursery|primary|secondary] parent_id: Many2one(res.partner) ← payer is_sibling: computed → activates discount # Dynamic attrs <field name="vaccination_book" attrs="{'invisible':[('level','!=','nursery')]}"/> <field name="test_score" attrs="{'required':[('level','=','secondary')]}"/> # Prorated = months_studied / total × fee
Business Impact

Delegation inheritance eliminated contact duplication for multi-child families. Dynamic UI reduced form errors. Sibling discount automation removed spreadsheet lookup steps. Payment validation prevents enrollment without financial clearance. Prorated fee calculation handled withdrawals fairly.

04
Omnichannel Sales & POS
Sales · POS · eCommerce · Exchange Logic
Omnichannel
Master Data (SKU/Barcode)→ POS + Online→ Unified Stock→ Exchange / Return→ Revenue Reconciliation
Problem

Retailer with webshop + physical POS had no unified stock. Return refunds used current prices instead of original transaction price, causing accounting disputes. Pre-orders had no tracking. POS errors had no structured correction process.

Solution

Unified Sales + eCommerce + POS on single inventory. Return logic locks refund to original invoice price. Pre-order uses deposit + stock reservation. Error transaction creates a revenue-correction sale without re-deducting stock.

Key Features
  • SKU, ISBN, Barcode + Product Variants (Size, Color)
  • Separate tax rules per product category
  • POS: barcode scan → instant stock + e-invoice
  • Pre-order: deposit → reserve → arrival alert
  • Exchange: new_price > orig → additional invoice
  • Exchange: same price → stock swap, no invoice
  • Error: cancel + new sale, no double stock deduction
Exchange & Error Transaction Logic
# Return: price locked to original invoice original_price = pos_order_line.price_unit # NOT product.lst_price (prevents price drift) refund_amount = original_price × return_qty # Exchange price comparison price_diff = new_price - original_price if price_diff > 0: create_additional_invoice(price_diff) if price_diff == 0: stock_swap_only() # no invoice # Pre-order state machine draft → deposit_paid → reserved → fulfilled # Error correction: revenue only, no stock cancel_error_invoice() new_sale(stock_move=False) # revenue-only
Business Impact

Unified inventory eliminated weekly manual reconciliation between channels. Return accuracy removed the main source of accounting disputes. Pre-order system reduced stockout-related lost sales. Error correction replaced ad-hoc cash adjustments with a traceable, auditable process.

05
Transit Inventory System — In-Transit Warehouse Hub
Inventory · Logistics · State Machine · Cron Automation · Exception Handling
Inventory
Warehouse A→ STR Approval→ Wizard Popup→ Transit Hub (Virtual)→ ASN Email→ Warehouse B→ Done / Discrepancy
Problem

North-South shipments take 3–5 days. High-value inventory "disappeared" from the system during transit — no visibility, no accountability between warehouse keepers, no structured handling for damaged or missing items on arrival. Discrepancies had nowhere to go.

Solution

stock.transfer.request (STR) model with full state machine. Transit Hub as virtual warehouse maintains stock balance visibility at all times. ASN auto-email on shipping. Cron job escalation for overdue transfers at 23:00. Wizard popup enforces condition declaration before dispatch commitment.

Key Features
  • STR: Draft → To Approve → Approved → Shipping → Done/Discrepancy
  • Source stock ↓ → Transit ↑ → Destination ↑ on receipt
  • ASN email auto-triggers on state = "Shipping"
  • Cron 23:00: scan scheduled_date + 5/7/10 day alerts
  • Wizard popup: "No issues" OR select issue type (required)
  • Issue types in Inventory > Configuration > Transfer Issues
  • Partial receipt → Discrepancy state → accounting compensation
State Machine, Cron & Exception Handling
# STR State Machine Draft → To Approve → Approved → Shipping → Done | Discrepancy # Transit location balance WHA → TRANSIT: qty out of WHA qty into TRANSIT TRANSIT → WHB: qty out of TRANSIT qty into WHB # Partial receipt → Discrepancy state
# ASN trigger on state write on_write(state='shipping'): send_mail(asn_template_id) # Cron 23:00 — escalation def _cron_transit_alert(self): overdue = self.search([ ('state','=','shipping'), ('sched_date','<', today-5) ]) overdue.send_escalation_email() # Wizard popup validation if not wizard.no_issues: if not wizard.issue_type_id: raise ValidationError( "Select issue type or confirm OK" )
Business Impact

Eliminated the in-transit "black hole" — logistics teams have live stock visibility between all warehouses. ASN automation reduced Warehouse B preparation time. Cron escalation prevents goods sitting unprocessed for days unnoticed. Wizard popup created a clear responsibility handoff — warehouse keepers must formally commit to condition before dispatch. Discrepancy state ensures every variance is recorded and assigned for accounting compensation, improving inventory accuracy network-wide.

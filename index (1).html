Agile vs Waterfall · Go-Live Strategies

Waterfall is preferred for ERP implementations with clear, stable requirements and hard go-live dates. Agile/Scrum works well for module-by-module rollouts where requirements evolve.

Go-live strategy selection depends on risk tolerance, data complexity, and business continuity requirements:

Big Bang Phased Rollout Parallel Run

Big Bang — All modules live simultaneously · High risk, fastest ROI
Phased — Module-by-module · Balanced risk, incremental adoption
Parallel — Old + new systems run together · Lowest risk, highest cost

3-Tier Architecture

Odoo follows a strict three-tier model that separates data persistence, business logic, and presentation — enabling scalability and clean module boundaries.

Layer 1: PostgreSQL
→ Relational data, ORM tables

Layer 2: Python (ORM + Logic)
→ Models, methods, wizards

Layer 3: OWL / XML Views
→ Forms, lists, kanban, QWeb
🏗️
MVC Design Pattern

Odoo's architecture maps cleanly to Model-View-Controller. Understanding this pattern is essential for writing accurate functional specifications that developers can implement directly.

Model → Python classes (models.Model)
ORM fields, business methods

View → XML templates (forms, lists)
QWeb reports, OWL components

Controller → HTTP routes, JSON-RPC
Request/response handling
🔗
Delegation Inheritance (_inherits)

Odoo supports three inheritance modes. Delegation inheritance (_inherits) creates a parent-child table relationship — the child record delegates field access to the parent via a foreign key. Used in Education CRM for Student → res.partner relationship.

# _inherit: extend same model
# _inherits: table delegation

_inherits = {
'res.partner': 'partner_id'
}
# student.record → partner fields
# via FK, no data duplication

Security — 4 Layers

Odoo's security model is hierarchical. A user must pass all four layers to access or modify a record. Understanding this is critical for designing access control in multi-department ERP deployments.

CRUD Rights (Access Control List)
Create / Read / Write / Delete per model per group
Record Rules (Domain Filter)
Row-level restriction — "only own department records"
Menu Visibility
Groups control which menu items are visible
View / Field Access
Field-level invisibility, readonly, required by group

01
Procurement & Dynamic Approval
Purchase · Inventory · QC
Purchase
Purchase Request→ Approval Engine→ Purchase Order→ Receipt + QC→ Inventory Update
Problem

Procurement approvals were manual and inconsistent across departments. Purchase Orders were created without budget validation or manager sign-off — resulting in unauthorized spending, duplicate orders, and inventory discrepancies. No audit trail existed.

Solution

Designed a dynamic multi-level approval matrix within Odoo Purchase. Approval routing adapts automatically based on purchase amount threshold and department. QC integration validates items before inventory update triggers.

Key Features
  • PR → PO flow with state tracking
  • Amount-based routing (L1/L2/L3 approval)
  • Department-specific approval matrix
  • QC pass/fail before stock update
  • Rejection with mandatory comment
  • Landed cost auto-allocation on receipt
Database Design
purchase.request id, name, department_id → hr.department approver_ids → res.users (M2M) state: draft|submitted|l1_approved|l2_approved|done|rejected total_amount: Monetary ← triggers level routing purchase.request.line request_id → purchase.request (M2O) product_id → product.product qty_requested, qty_received qc_status: pending|pass|fail
Business Impact

Eliminated unauthorized procurement by enforcing approval gates before any budget commitment. Full audit trail from PR creation through receipt. QC gate prevents defective goods from entering inventory — reducing write-offs. The entire PR-to-receipt flow is traceable in a single Odoo chain with zero manual handoffs.

02
B2B Master Contract Management
Sales · Invoicing · Delivery
B2B Sales
Master Contract→ Sub-Orders→ Partial Delivery→ Delivery Invoice→ Contract Utilization
Problem

Enterprise clients negotiated annual contracts with multiple delivery releases spread across the year. No system tracked how much of a contract had been delivered vs. remaining. Invoicing was manual, error-prone, and often over or under-charged customers.

Solution

Designed a Master Order + sub-order architecture. The master contract holds committed volumes and locked pricing. Sub-orders pull against the master contract. Computed fields update delivered/remaining in real time. Invoices trigger only on confirmed delivery.

Key Features
  • Master contract → sub-order linkage
  • Computed remaining_qty per product
  • Delivery-triggered invoice generation
  • Price lock from master agreement
  • Over-delivery validation constraint
  • Contract utilization dashboard widget
Key Logic — Computed Fields & Validation
# Computed remaining quantity sale.master.contract.line: committed_qty: Float delivered_qty = fields.Float( compute='_compute_delivered', store=True ) remaining_qty = committed_qty - delivered_qty # Validation: prevent over-delivery @api.constrains('delivered_qty') def _check_qty(self): if self.delivered_qty > self.committed_qty: raise ValidationError("Exceeds contract volume") # Invoice trigger on delivery invoice_policy = 'delivery' ← on sale.order
Business Impact

Sales and finance teams gained real-time contract fulfillment visibility for the first time. Eliminated over-invoicing risk by tying invoice generation directly to confirmed deliveries. Account managers can proactively alert clients when approaching contract limits — reducing disputes and improving client trust.

03
Education CRM — Admissions System
CRM · Custom Module · Financial Logic
CRM / Custom
Parent Lead→ Student Profile→ Level Assessment→ Sibling Discount→ Enrollment
Problem

An education center needed a CRM for admissions where the parent is the payer (customer) but each child is the enrolled student. Standard CRM had no parent-child data relationship, no dynamic UI by education level, and no automatic sibling discount logic.

Solution

Used Odoo Delegation Inheritance (_inherits) to model Student as an extension of res.partner — inheriting all contact data via FK. Dynamic UI rendered conditionally by education level using attrs. Financial logic auto-calculates sibling discounts and validates payment status before enrollment confirmation.

Key Features
  • Delegation inheritance (Student → res.partner)
  • Dynamic UI by level (Nursery/Primary/Secondary)
  • Sibling discount auto-calculation
  • Payment check before enrollment confirm
  • Admissions pipeline in Kanban view
  • Academic term tracking per student
Data Structure & Dynamic UI Logic
education.student _inherits = {'res.partner': 'partner_id'} # Inherits: name, phone, email, address education_level: Selection[nursery, primary, secondary] parent_id: Many2one(res.partner) ← payer sibling_ids: One2many(education.student) # Dynamic UI (attrs by level) <field name="subject_stream" attrs="{'invisible':[('level','!=','secondary')]}"/> # Sibling discount logic sibling_discount = len(sibling_ids) * 5% ← max 15% # Payment validation constraint @api.constrains('state') def _check_payment(self): if self.state == 'enrolled' and not self.payment_cleared: raise ValidationError("Payment required before enrollment")
Business Impact

The _inherits architecture eliminated data duplication between student and contact records — one update flows through both systems. Dynamic UI reduced form errors (admissions staff no longer see irrelevant fields). Sibling discount automation removed a manual step that previously required spreadsheet lookup, cutting per-family enrollment processing time significantly.

04
Omnichannel Sales & POS
Sales · POS · eCommerce · Returns
Omnichannel
SKU / Barcode Setup→ Online + POS Sale→ Unified Stock→ Exchange / Return→ Revenue Reconciliation
Problem

A retailer with both a webshop and physical POS had no unified inventory view. Returns at POS used current prices instead of original purchase prices — causing refund discrepancies. Pre-orders had no systematic handling, and error transactions lacked a correction workflow.

Solution

Integrated Odoo Sales, eCommerce, and POS into a single inventory source of truth. Designed return logic that locks refund amount to the original invoice price. Pre-order flow uses deposit payment with stock reservation. Error transaction handling with manager override and correction journal entry.

Key Features
  • Master data: SKU, barcode, variant matrix
  • Shared real-time stock across all channels
  • Original-price lock on returns/exchanges
  • Pre-order: deposit → reservation → fulfill
  • Price difference handling on exchange
  • Error transaction correction workflow
  • Cross-channel revenue dashboard
Exchange & Pre-Order Logic
# Return/Exchange: price locked to original original_price = pos_order_line.price_unit # NOT product.lst_price (current price) refund_amount = original_price * return_qty price_diff = current_price - original_price # if exchange: customer pays/receives diff # Pre-order state machine draft → deposit_paid → stock_reserved → delivery_ready → fulfilled # Error transaction: manager correction error_transaction.state = 'flagged' manager.approve() → correction_entry # audit log preserved, not deleted
Business Impact

Unified inventory eliminated weekly manual stock reconciliation between channels. Return price accuracy removed a significant source of accounting disputes — all refunds now precisely match original transaction amounts. Pre-order system reduced stockout-related lost sales by enabling reservations. Error correction workflow replaced ad-hoc cash drawer adjustments with a traceable, auditable process.

05
Transit Inventory System — Multi-Warehouse Hub
Inventory · Logistics · Automation · Exception Handling
Inventory
Warehouse A (Source)→ Transit Hub (Virtual)→ ASN Email Auto→ Warehouse B (Destination)→ Discrepancy Handling→ Reconciliation
Problem

A company with multiple warehouses needed goods to pass through a central transit hub before reaching their destination. In-transit stock was invisible in the system — creating a "black hole" between dispatch and receipt. Discrepancies (damaged/missing items) had no structured handling process, leading to untracked inventory losses.

Solution

Configured Odoo multi-step warehouse routing with Transit Location as a virtual intermediate warehouse. ASN (Advance Shipping Notice) emails auto-trigger when goods leave the source warehouse. A full state machine governs the transfer lifecycle. Discrepancy workflow allows partial receipt with automatic backorder creation and variance logging for audit.

Key Features
  • A → Transit → B three-step routing
  • Virtual transit location (always in balance)
  • Automated ASN email on dispatch confirmation
  • Real-time in-transit stock visibility
  • Discrepancy: partial receipt + backorder
  • Cron job: escalation email if in-transit > X days
  • Transfer reconciliation report by period
State Machine & Cron Automation
# State machine: stock.picking (transfer) draft → confirmed → assigned → in_transit → partially_received → done | discrepancy # Transit location: always zero net balance WHA → TRANSIT: move qty OUT of WHA INTO TRANSIT TRANSIT → WHB: move qty OUT of TRANSIT INTO WHB # If WHB partial receipt → backorder auto-created # Automated ASN email trigger ir.actions.server → send_mail(template_id) trigger: on write, state = 'in_transit' # Cron: escalation if stuck in transit @api.model def _cron_transit_alert(self): overdue = self.search([ ('state','=','in_transit'), ('scheduled_date','<', today - 3 days) ]) overdue.mapped('send_escalation_email')() # Discrepancy variance log expected_qty vs received_qty → variance_line variance_line.responsible_id = warehouse_manager
Business Impact

Eliminated the "in-transit black hole" — logistics teams now have live visibility of all stock between warehouses at any moment. ASN automation reduced Warehouse B preparation time significantly. The cron escalation prevented goods from sitting unprocessed in transit for days unnoticed. Discrepancy handling ensured every variance is recorded and assigned — improving inventory accuracy across the entire distribution network and enabling accurate financial valuation of in-transit goods.
